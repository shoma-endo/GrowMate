## 1. 要約：プロダクト思想・背景

### 1.1 メディア運用と AI の役割

- **内部リンクを張り巡らせてドメイン全体の強度を上げる**ことに強い期待を置いており、手作業では負荷が高い領域を **登録済みコンテキストを読ませた AI** に担わせたい、という動機が繰り返し表れている（過去記事・固定ページ・お客さまの声ページ等の参照）。
- **LP・記事の型（見出し階層・文字数・箇条書きなど）を厳密に守らせる**ニーズがある。プロンプトで「最優先・違反不可」レベルまで指定している例がある。

### 1.2 品質・データ完整性

- **エビデンス（根拠）チェックを最優先課題**として位置づけ、項目の最大数とコスト感を踏まえて **優先順位付け**したい、という発言がある。
- 分析系では **取得できる指標はできるだけ保持**し、UI では表示のオンオフを選べる形がよい、という考え方がある。**順位だけでなくクリック率なども重要**、**上書きより履歴が欲しい**（サーバ負荷とのトレードオフ懸念あり）という技術的関心がある。

### 1.3 WordPress 連携の目的

- **記事修正の現実的なフロー**として、従来のツール内保存だけでは「修正後に公開した記事を再編集しづらい」問題があり、**WordPress から読み込んでツールで編集**したい、という目的説明がある。
- **ツール→WP 一方向同期**だと WP 側で加えた装飾（色・画像等）を活かした修正が難しくなる懸念があり、**キャンバスをそのまま保存しやすい形**の方が望ましい、という UX 判断が述べられている。

### 1.4 ブログ作成フローと「目的」中心のコミュニケーション

- **本文を一気に出すと API コストと手間が大きい**ので、**1 見出し＋本文ずつ**出す必要がある、という制約認識がある。
- 「ステップの流れ」という言い方は誤解を招くため避け、**目的（本文が 1 見出しずつ出ること）が達成されればよい**、というコミュニケーション上の指摘がある。
- ブログ作成の各ステップの**文言（ラベル）を具体的に定義**し、優先開発候補（ログイン追加・評価まわり・ブログアプデ・Instagram 等）について **工数見積りに基づき優先順位を変えたい**、という運用がある。

### 1.5 優先機能ロードマップ（発言ベース）

時期により表現は変わるが、チャット上では以下が「やりたいこと」として繰り返し列挙されている。

- 複数事業者（複数ドメイン）登録
- 広告文の複数案提案
- ブログ作成アプデ（カニバリ・キーワードギャップ等の前段、Gemini 利用希望の記述あり）
- Instagram 連携
- 利用制限（料金プランに応じた利用者数制限）
- メール・LINE ログイン、コンテンツ評価の初回評価挙動変更、など（別スレッドで工数前提の優先順位調整の記述あり）

### 1.6 協働・ガバナンス（開発との関係性）

- **勝手な仕様判断で前提が崩れること**への強い警戒がある。**事前許可なく挙動を変えない**こと、**ミスコミュニケーションを防ぐため具体箇所への差し込み確認**を求める姿勢がある。
- 認識齟齬が **余計な工数とストレス**につながる、という感情表出がある（再作業コストへの嫌悪）。
- **チャット（メッセージ）UI とキャンバスの役割分担**をはっきりさせたい。**メッセージ表示に操作を集約**し、**キャンバスは文章フィードバックに集中**。「あっちもこっちも」だと UI が悪化する、という設計思想がある。
- ユーザー視点では **途中で挙動が変わると混乱**するので、**これまで通りのステップ感**を維持する方がわかりやすい、というフィードバックがある。

### 1.7 入力規模とロバストネス

- キーワード数が少ないときは問題ないが、**数十件規模のキーワード投入で挙動が崩れる**（意図しないキーワード混入等）という報告と、**クロード側では期待通り**という対比がある。ツール側のプロンプト／処理の堅牢性が課題として上がっている。

## 4. 再抽出手順（開発者向け）

同一条件で JSON を取り直す場合:

```bash
lark-cli auth status
lark-cli im +messages-search \
  --chat-id oc_b695d86123122d08fff4612843fd2a86 \
  --sender ou_1effb2a8cb3ac279cfb6cec370953e4b \
  --page-all \
  --format json > /tmp/kaoru-messages.json
```

検索 API の都合で **全履歴が常に返る保証はない**。件数・日時レンジは再抽出のたびにメタ情報を更新すること。
