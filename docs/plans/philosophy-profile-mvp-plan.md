# 哲学プロファイル機能 実装可否判断 ＆ 段階的ロードマップ

## Context（なぜ）
クライアントは、ブログ/LP/広告生成の出力を「本人らしく」するため、ユーザーごとの哲学プロファイルを
生成・保持し、各生成リクエストに注入する「記憶レイヤー」を求めている（仕様2ファイル）。
本ドキュメントは「実装可能か」の俯瞰判断と、実装する場合の段階的ロードマップを示す。

**元仕様**: `docs/specs/growmate_philosophy_system_spec.md`（実装仕様書）／`docs/specs/growmate_philosophy_interview_v2.md`（質問・選択肢・重み）。

## 結論
**実装可能。GrowMate の既存パターン（JSONB プロファイル保存・動的 system prompt 注入・LLM 共用エンジン・
外部設定の管理画面）の拡張で実現でき、アーキテクチャ変更は不要。**
ただし単発作業ではなく複数週規模。仕様§13に従い段階導入する。

**クライアント合意（先行）**: まず **パイロット（P0・軽量実装）** で効果を検証する。本書の Phase 0〜5（フル実装）はパイロット結果を見て着手する。

## 先行実装：パイロット P0（クライアント合意・最優先）

### 方針（合意内容）

1. **外部で哲学診断**（専用 GPTs 等）を実施し、出力テキストをユーザーが **コピペで保存**する。
2. 保存先は既存の事業者情報 **サービス内容（5W2H）＞ `strength`（UI 表示名のみ変更）**。
3. **この状態で毎回生成に読み込めれば、それでよし**（新 DB・専用注入開発は当面不要）。
4. **別フィールド／別テーブルが必要になった時点で**、その時に別保存の仕組みを構築する（先回りしない）。

### なぜ成立するか

- `briefs.data.services[].strength` は既に `{{strength}}` として広告・LP・ブログ系プロンプトに注入済み（`PromptService.buildServiceVariables` → `replaceTemplateVariables`）。
- パイロットは **UI・運用・文言の整備** が中心。データモデル変更・マイグレーションは不要。

### P0 スコープ（GrowMate 側）

| 区分 | 内容 | やる / やらない |
|------|------|----------------|
| UI | ラベル `強み` → **`強み+毎回反映したい内容`** | ✅ |
| UI | 説明文・placeholder・文字数目安（例：800字以内） | ✅ |
| UI | 貼り付けテンプレ（6項目）の折りたたみ表示 | ✅ 推奨 |
| UI | 外部診断（GPTs）URL の案内リンク | ✅ 任意（URL 確定後） |
| データ | `strength` キー・`briefs` テーブルは **現状維持** | ✅ 変更なし |
| 診断 | GrowMate 内の10問 UI・LLM 生成 | ❌ 当面不要 |
| 注入 | `getSystemPrompt` 等への哲学専用注入 | ❌ 当面不要（既存経路で足りる） |
| DB | `philosophy_profile` / `edit_log` 等 | ❌ 当面不要 |

**代表ファイル（実装時）**: `app/business-info/components/ServiceCard.tsx`、`src/lib/prompt-descriptions.ts`（`strength` の説明文言）

### P0 スコープ（運用・社外）

| 区分 | 内容 |
|------|------|
| 診断 | 専用 **GPTs**（`interview_v2` ベース・対話後に貼り付け用テキスト出力） |
| 出力形式 | 800字以内・6項目テンプレ（譲れない価値観／やらないこと／トレードオフ／トーン／読者像／語り口） |
| ユーザーフロー | GPTs で診断 → 出力をコピー → 事業者情報の当該欄に貼付 → 保存 |

### P0 工数

| 項目 | 理想人日 |
|------|----------|
| GrowMate（ラベル・説明・テンプレ・リンク） | **0.5〜1日** |
| GPTs 作成（社外・カオルさん側） | **約半日**（本計画の開発工数外） |

### P0 完了判定

- 事業者情報で「強み+毎回反映したい内容」に哲学テキストを保存できる。
- ブログ／LP／広告の生成で、欄の内容がプロンプトに載っている（既存 `{{strength}}` 経路）。
- 同一サービスで、欄あり／なしで出力に差が出ることを手動確認。

### パイロット後の分岐（別保存は必要時のみ）

| 状況 | 次の一手 |
|------|----------|
| `strength` 1欄で運用・品質とも十分 | フル Phase 0〜5 は据え置き or 長期バックログ |
| 強み（事実）と哲学（価値観）の **分離** が必要 | `briefs` に新フィールド追加（例：`philosophy_reflection`）＋プロンプト変数追加。**小規模（2〜3人日）** |
| 構造化プロファイル・学習ループが必要 | 本書 Phase 0〜5（フル実装）へ |

> **鉄則**: 別保存は「必要になったら作る」。P0 では `strength` への同居を許容する（サービスの強み＋毎回反映したい哲学を1欄に書く運用）。

## 既存資産マッピング（要点）
- 外部設定: `prompt_templates`(JSONB)＋`/admin/prompts` 管理画面 → `question_definition`/`*_config` の前例
- ホット層保存: `briefs.data`(JSONB)＋`upsert_brief` RPC＋Zod → `philosophy_profile` の保存パターン
- 注入点: `app/api/chat/anthropic/stream/route.ts` の `getSystemPrompt()`、`src/server/services/promptService.ts` の `replaceVariables()`
- LLM 共用: `src/server/services/llmService.ts` `llmChat()`（再生成・差分分類・Layer3 提案に流用）
- モデル定義: `src/lib/constants.ts` `MODEL_CONFIGS`
- 編集UI足場: `app/chat/components/CanvasPanel.tsx`（Canvas編集フロー。哲学向け edit_type 計装は Phase 3 で新規）

## リスク／難所
1. **edit_log 収集（最難所）**: UI操作から before/after・edit_type を体系記録する計装が侵襲的。
2. **注入点の多重性**: ブログ7ステップ＋Canvas＋広告＋LP（**GSC/Google Ads 提案は対象外**）。注入箇所を§6.3に沿って絞る設計判断が必要。
3. **規模**: Phase 0〜5（＋0+・2+ 補完）。フェーズ毎の段階リリース必須。

## コンテキストウィンドウ／注入の実装前提（必須遵守）
ベースモデル `claude-opus-4-7`（約200K入力）。system は単一 `cache_control: ephemeral` ブロック
（`app/api/chat/anthropic/stream/route.ts:257-262`）。step7 は `max_tokens: 64000`。
2層モデル（ホット注入／コールド非注入）によりコンテキストは構造上肥大しないが、以下を遵守する。
1. **§6.3の断片注入を厳守**: profile 全文を投げず generation_type 別に必要部分のみ注入し小さく保つ。
2. **step7 のヘッドルーム実測**: 出力64K予約＋既存（ブログprompt＋brief＋step1〜6履歴）に上乗せされる最もタイトなケースで長尺セッションの溢れを実測確認。
3. **再生成入力のウィンドウ化（唯一の青天井ポイント）**: Phase 4 の再生成LLM呼び出しは入力＝現profile＋未消費 edit_log 集約。**1回の再生成で投入する edit 件数に上限を設ける（直近N件/バッチ）**。trigger_config 等で外部設定化する。
4. **キャッシュ配置**: 安定部分（brief・ブログ用prompt・profile）をキャッシュ前段、リクエスト毎に変わる内容を後段に置き、profile 変更（本人承認制の再生成時のみ＝低頻度）以外でキャッシュを切らさない。

## MVP の定義

### パイロット P0（当面のリリース＝クライアント合意）

**「外部診断の結果を既存 `strength` に保存し、毎回の生成に反映する」**まで。上記 **先行実装：パイロット P0** を参照。

### フル MVP（R1・パイロット検証後に検討）

**「哲学プロファイルを生成し、生成リクエストに注入して、出力を本人らしくする」**までをフル MVP とする。
学習ループ（修正ログ収集・分類・再生成・追い質問・Layer3）は**ポストMVPへ全面後回し**。
理由: 再生成・追い質問は edit_log が閾値（意味ある修正10件）まで溜まって初めて発火し、リリース直後は動かないため、
フル MVP に含めても当面価値が出ない（仕様§13「初動必須3点以外は段階追加可」とも整合）。

**P0 との関係**: P0 で仮説が確認できた後に着手。P0 成功時も GrowMate 内簡易診断（5〜7人日）を挟む選択肢あり。

### リリース1（R1）スコープ = Phase 0 + 0+ + 1 + 2 + 2+（フル・未着手）

（Phase 0+1+2 のみを「最小 MVP」と呼ぶ場合は 17〜26 理想人日。実用完了は **2+ 含む R1 = 20〜32 理想人日**を推奨）

#### Phase 0（基盤・後戻り不能を回避）
- `question_definition` を **バージョン付きID**（`Q3_v1`／`Q3_v1_opt1`）で外部設定化（DBテーブル＋10問seed）
- `weight_config`（スコアリング重み）を外部設定化（コード埋め込み禁止）
- `philosophy_profile` テーブル＋`upsert` RPC（briefs 同型）
- `philosophy_onboarding_answers` テーブル＋**初回オンボ用の基本 upsert RPC**（生回答の永続化のみ。profile 導出は Phase 1）
- `edit_log` テーブル**定義のみ**新設（is_meaningful フラグ含む）。**収集は Phase 3 まで後回しだが §13-3 の後戻り不能回避のためスキーマだけ先に用意**
- Zod スキーマ: `philosophy_profile` / `question_definition` / `philosophy_onboarding_answers` の型定義
- 代表ファイル: `supabase/migrations/`（新規）, `src/server/schemas/`（新規）
- **Phase 4 まで除外**: `trigger_config`（再生成トリガー専用）

#### Phase 0+（生回答の再オンボ・部分再計算）
- **Phase 0 との責務分離**: Phase 0＝テーブル定義＋初回保存 RPC／Phase 0+＝再オンボ・特定問のみ修正・バージョン共存の**アプリ層ロジック**
- 全問やり直し（再オンボ）時の回答上書きと `philosophy_profile` 全件再計算
- 特定問のみ修正時の部分再計算（旧 `question_id` バージョンとの共存は §8.3 どおり）
- 再オンボ・部分更新用の Server Action / RPC（Phase 1 の初回フローとは別エントリ）

#### Phase 1（オンボーディング→プロファイル生成）
- オンボーディング10問UI（`/business-info` 系の前例を踏襲、1問ずつ提示・リカバリー可）
- スコア合算→正規化→`thinking_type.scores`/`dominant`/`deficit` 算出（純TS）
- §5.2 クロスチェック（Q3×Q4, Q6×Q9）で**各 `core_values` 項目の `confidence`（high/medium/low）**を判定
- プロファイルは **`overall_confidence: low`**（v2_initial の初期値・§4.3/§5.1）で保存（`upsert` RPC）。※項目別 `core_values[].confidence` はクロスチェックで high になり得る＝**プロファイル全体の `overall_confidence: low` とは別レイヤー**（後述 X3 参照。初版は使用後の修正ログで全体確信度を引き上げる二段構え）
- API: `GET /onboarding/questions`, `POST /onboarding/answers`

#### Phase 2（注入・初回生成が本人らしくなる）
- `getSystemPrompt()`（`src/lib/prompts.ts:944`）に profile 断片注入を追加（`anthropic/stream/route.ts` 経由の**初回生成**）
- model キーで §6.3 の type別断片を出し分け（全タイプ共通＝core_values/voice_fingerprint/thinking_type、広告・LP＝enemies/ideal_persona 追加、ブログ＝tradeoff_tendencies 追加）
- 注入強度を generation_type ごとに調整可能化（既定値は"説教臭くならない"レベル）
- **哲学（思想・内容）とフォーマット（構造）は直交し両立**。既存の厳守ルール（広告=全角15文字/kw出現制限 `prompts.ts:85-88,160`、LP=16パート構成 `prompts.ts:624-633`）は器として維持し、その中で内容・トーンに効かせる
- **対象＝ブログ/LP/広告文の3タイプすべて**（仕様§1）。GSC/Google Ads提案は対象外

#### Phase 2+（Canvas 注入・実用補完）
- `app/api/chat/canvas/stream/route.ts` の独自 system prompt にも profile 断片を注入（`getSystemPrompt` は通らない）
- ブログの Canvas 再編集時も本人らしさを維持する。**リリース1の完了条件に含める**（Phase 2 のみでは体感価値が抜ける）

## ポストMVP（フェーズ別・段階リリース）

フルスコープは規模が大きいため、**フェーズ単位でリリース**する。各フェーズ完了ごとにクライアント確認・QAを挟む。

### Phase 3: 修正ログ収集・分類（仕様§7）
- `edit_log` への追記（`POST /edits` 相当）、操作種別ベースの `edit_type` / `is_meaningful` 分類
- **A1** 書き込みサービス・API
- **A2** Canvas 計装（before/after・edit_type）— **最難所**（Phase 2+ と `canvas/stream` を共有する作業）
- **A3** ブログチャット修正・LP/広告側の計装（経路ごとに薄く）
- **A4** 意味あり/ノイズ判定（操作種別のみ。差分LLM分類は任意で +2〜3日・運用コスト増）
- ※テーブルは Phase 0 で用意済み。ここから記録開始

### Phase 4: 再生成・追い質問（仕様§8/§9）
- **B1** `trigger_config` DB/seed、§8.1 AND条件の閾値判定
- **B2** 再生成 LLM（現 profile ＋未消費 edit 集約。**投入 edit 件数に上限＝ウィンドウ化**）
- **B3** 再生成案提示・承認/却下 UI（`profile/regenerate`, `profile/confirm` 相当）。**本人合意なしに反映しない**
- **B4** confidence 昇格、`consumed_by_regen` / `edits_since_last_regen` 更新
- **B5** 追い質問（`followup/next` 相当・単発1問・回答反映）

### Phase 5: Layer3 不足軸レンズ提案（仕様§10・任意）
- `deficit` に対応する固定レンズで新哲学を仮説提示（採用/棄却/修正）
- edit_log 蓄積に依存しない（Phase 1 で `deficit` は出る）。Phase 4 完了後でも単独リリース可

## 鉄則（実装時に必ず守る）
- 本人合意なしにプロファイルを書き換えない（§8.3）
- weight_config 変更で既存プロファイルを再計算しない（凍結デフォルト）
- 質問変更は新バージョンID追加、旧回答は共存
- エマジェネティクス診断名をUI表示しない（内部は analytical/structural/social/conceptual）

## オンボーディング設計の確定事項（着手前に確定）

質問文・選択肢・4色重みは `growmate_philosophy_interview_v2.md` で明確。一方、回答→プロファイル変換ロジックに
未確定があり、特に「後から振れない」ものは着手前に固める。

### 不変なもの／変更可能なものの切り分け
- **不変＝選択肢定義（option_id）**: 全ユーザー共通マスタ。意味の固定。振り直し禁止。
- **変更可能＝ユーザーの回答**: いつでも再回答・修正可。リカバリー前提。
- **導出物＝プロファイル**: 回答から再計算・本人主導で再生成可。

### option_id 採番規約（§13-1 初動必須・後戻り不能）
- 形式: `Q{n}_v{ver}_opt{m}`（例 `Q3_v1_opt1`）。質問IDもバージョン付き（`Q3_v1`）。
- 連番方式。**欠番許容・番号の振り直し禁止**（過去回答が別選択肢を指すのを防ぐ）。
- 選択肢の実質変更・追加・重み変更は**質問ごと新バージョン**（`Q3_v2`）。旧版の過去回答は共存（§8.3）。
- クライアントには「このID体系で固定」と一報すれば足りる（意思決定不要・実装側で規約定義）。

### 回答のリカバリー設計
- オンボーディング確定前: 前問へ戻る・選び直し（保存前なので影響なし）。
- 確定後: 再オンボーディング（全やり直し）／特定問のみ修正（部分再計算）を許可。本人主導の更新＝§8.3 鉄則に反しない。

### ★ 2仕様間の不整合（着手前に解消・I1/I3/I4で吸収）
- **X1**: `enemies` の source がシステム仕様§4.3＝`["Q4_v1"]`（Q4のみ）／インタビュー出力例＝`["Q4","Q8"]`（Q4+Q8）で食い違い → I4 で対応表を確定。
- **X2**: source の表記が `Q3_v1`（バージョン付き）と `Q3`（無し）で混在 → I1 のID体系で統一。
- **X3**: 項目別 high/medium/low から `overall_confidence` を出す規則が両書とも未定義 → I3。
- **X4**: §5.2 は high と low のみ定義し `medium` の付与条件が無いのに出力例で使用 → I3。

## 検証方法

共通: 各 Phase 後に `npm run lint` / `npm run build` / `npm run knip`（quality-gate スキル準拠）

| Phase | 検証内容 |
|---|---|
| 0 | マイグレーション適用、seed 10問の ID 体系、初回生回答 upsert の手動確認 |
| 0+ | 再オンボ・特定問のみ修正→profile 全件/部分再計算の手動確認 |
| 1 | 10問回答→スコア正規化・confidence 判定（I3 確定式どおり） |
| 2 | blog/LP/広告それぞれで profile 注入あり/なしの出力差 |
| 2+ | Canvas 編集で profile 注入あり/なしの出力差 |
| 3 | 編集操作ごとに `edit_log` が追記され `is_meaningful` が期待どおり |
| 4 | 閾値到達→再生成案提示→承認時のみ profile 更新。追い質問 1 問の反映 |
| 5 | deficit に応じたレンズ提案・採用/棄却の記録 |

**R1 完了判定**: オンボ → プロファイル保存 → blog/LP/広告の初回生成 **および** Canvas 編集に注入され出力が変化、までを一気通貫で確認。

**R2 完了判定**: 意味ある修正の蓄積 → 再生成案 → 本人承認 → profile 更新 → 以降の生成に反映。

**フル完了判定（R3 含む）**: 上記に加え Layer3 提案の採用/棄却が edit 学習と整合すること。
