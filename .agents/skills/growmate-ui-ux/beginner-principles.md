# 初心者向け UX 原則（GrowMate）

広告運用・コンテンツマーケティング初心者が **初回から最後まで迷わず** 使える UI を設計するための原則。**UI 実装の開発時**（着手前・コーディング中）に参照する。

## 1. 用語と説明

| 原則 | 具体例 |
|------|--------|
| 専門用語は初出で補足 | 「CTR（クリック率）」のように括弧またはツールチップで説明 |
| 内部用語を表に出さない | API 名、DB 列名、診断型名等 |
| エラーは原因 + 直し方 | ❌「Invalid input」→ ✅「Google アカウントの連携が切れています。設定画面から再連携してください」 |
| 略語だけのボタン禁止 | ❌「OAuth」→ ✅「Google で連携する」 |

## 2. 情報設計

| 原則 | 具体例 |
|------|--------|
| 1 画面 1 主目的 | ダッシュボードに設定変更を混ぜない。設定は `/setup` 系へ |
| 1 画面 1 主 CTA | プライマリボタンは 1 つ。二次操作は outline / ghost |
| 段階的開示 | 高度なオプションは「詳細設定」アコーディオン内。初期は最小フィールド |
| 空状態に次の一手 | 「データがありません」だけで終わらせず、「GSC を連携する」リンクを置く |
| 比較しやすさ | 候補一覧はカード形式（合意済み仕様）。テーブルは分析画面に限定 |

## 3. フローと一貫性

| 原則 | 具体例 |
|------|--------|
| ステップ感を維持 | ブログ 7 ステップ等、既存フローの番号・ラベルを勝手に変えない |
| 予測可能な戻る | モーダル・ウィザードは閉じる/戻るが常に見える。ブラウザ戻るで破綻しない |
| 同種操作は同種 UI | 連携設定は `/setup/*` でパターン統一。各所で OAuth ボタン見た目を揃える |
| チャット vs キャンバス | 操作・ステップ進行 → チャット側。本文編集・リライト → キャンバス側 |
| 設定完了のフィードバック | 連携成功は toast + 状態バッジ更新。サイレント成功にしない |

## 4. フォームと入力

| 原則 | 具体例 |
|------|--------|
| ラベル必須 | placeholder のみをラベル代わりにしない |
| 入力例を helper text で | 「例: example.com/blog/post-slug」 |
| 破壊的操作は確認 | セッション削除、アカウント連携解除 |
| 大量入力の警告 | キーワード数十件投入時は件数表示・分割提案（§1.7 ロバストネス） |
| 保存状態の明示 | 保存中 / 保存済み / 未保存の区別。離脱時 confirm（未保存変更あり） |

## 5. 信頼と透明性

| 原則 | 具体例 |
|------|--------|
| AI の制約を隠さない | トークン上限で出力が分割される場合、ユーザーに説明して「続きを生成」を提示 |
| 処理中は止められる | 長時間 AI 処理中も UI を固めない。可能ならキャンセル |
| 自動変更は本人合意 | ユーザー設定・プロファイル等は勝手に上書きしない。確認 UI を挟む |
| 権限の理由を説明 | paid 機能へのアクセス制限時、「有料プランで利用可能」と理由を示す |

## 6. レスポンシブ

| 原則 | 具体例 |
|------|--------|
| モバイルファーストで設計 | 主要導線はスマホでも操作可能（セットアップ、ホームカード） |
| 複雑作業はデスクトップ想定 | チャット + キャンバス同時表示は PC 優先。モバイルは Sheet サイドバー等で代替 |
| 横スクロール回避 | 分析テーブルは列の優先度を付け、小画面では非表示列を検討 |
| 16px 以上の本文 | iOS 自動ズーム回避 |

## 7. アンチパターン（GrowMate で避ける）

- ダッシュボードに機能を詰め込みすぎる（初心者は何から触ればよいか分からない）
- マーケティングダッシュボード風の「映える」デザインで情報密度を上げる
- ツールチップなしの KPI 略語だらけの表
- モーダル on モーダル
- AI 生成中に空白画面（フィードバックなし）
- エラー時に「もう一度お試しください」のみ
- 新パレット・新フォントの無断導入
- キャンバスとチャットの両方に同じ操作ボタンを置く
