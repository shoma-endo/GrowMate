# Documentation Update Workflow

GrowMate の docs は公開サイトではなく **内部設計書体系**。分類ごとに役割が異なるため、更新前にどの分類に属するかを特定する。

## docs の分類と役割

| 分類 | 役割 | 更新タイミング |
|------|------|---------------|
| `docs/plans/` | 機能の設計書（実装前に作成しレビューを受ける正本） | 実装で設計と差分が出たら**同期更新必須** |
| `docs/specs/` | 合意済み仕様・外部 API セットアップ手順 | 仕様変更の合意後 |
| `docs/context/` | 背景知識・障害由来の知見（クライアント思想、LLM 設計、DB 上限） | 新たな知見・障害調査の確定後 |
| `docs/runbooks/` | 一回性の運用手順・バックフィル記録 | 運用作業の実施時 |
| `README.md` | プロジェクト概要・セットアップ・コマンド | コマンドや構成の変更時 |

## 手順

1. **変更範囲の把握**: `git diff --stat` でブランチ上の変更ファイルを洗い出す。
2. **影響 docs の特定**: 変更した機能に対応する設計書を探す（`grep -rl "<機能名・画面パス・テーブル名>" docs/`）。
   - 機能の挙動・UX・データフローの変更 → 対応する `docs/plans/*` 設計書
   - 外部 API の設定手順の変更 → `docs/specs/*`
   - 障害調査で得た横断的知見 → `docs/context/*`
3. **既存 docs を読んでから直す**: いきなり書き換えず、「何が古くなるのか／不足しているのか」を特定する。設計書のセクション構成・記法（設計判断の根拠を残すスタイル）を踏襲する。
4. **更新方針を提案して合意**: どのファイルをどう直すかを箇条書きで提示し、確認を得てから編集する。大きな書き換えはセクション単位で確認を挟む。
5. **検証**: リンク切れ（参照パスの実在）を確認する。スキル関連ファイルを触った場合は `npm run verify:agent-skills` を実行する。

## 設計書先行運用との同期（必須）

- 中〜大規模機能は **実装前に `docs/plans/` へ設計書を作成しレビューを受ける**（`agent-workflow-core` 参照）。
- 実装中に設計と異なる判断をした場合、コードだけ直して設計書を放置しない。**同じ PR 内で設計書を実態に合わせる**か、乖離をユーザーへ明示して更新の要否を確認する。
- 旧仕様の記述を残す場合は「変更履歴」または「旧仕様との違い」として明示し、現行仕様と混在させない。

## 最後のチェックリスト

- [ ] 変更したコードに対応する設計書・仕様書が更新されている
- [ ] 古い情報や誤解を招く記述が残っていない（設計書と実装の乖離ゼロ）
- [ ] 参照しているファイルパス・コマンドが実在する
- [ ] 新しい挙動・制約・設計判断の根拠が明記されている
