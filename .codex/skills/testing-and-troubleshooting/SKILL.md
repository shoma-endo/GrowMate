# 動作確認の基本方針

- 自動テストは未整備のため、**手動検証と `npm run lint` / `npm run build`** を基本とする。
- コード変更後は、可能な範囲で以下を行う:
  - `npm run lint` で静的チェック
  - `npm run build` でビルド確認
  - 関連画面の手動操作で挙動を確認

# 画面・機能ごとのチェックポイント

- **認証・Stripe 周り**
  - `app/page.tsx` から購入導線までの UI フローを一通り通して、認証状態とサブスクリプション状態に応じた表示切り替えが期待通りか確認する。
  - ログイン／ログアウト／トライアル／有料／利用停止など代表的なロールでの動作を意識する。

- **WordPress 連携変更時**
  - `/analytics` 画面で WordPress から取得した投稿が正しく表示されるか確認する。
  - `AnnotationPanel` での表示・保存動作（`content_annotations` への upsert）が正常に動くかを確認する。

- **GSC 連携変更時**
  - `/gsc-dashboard` で Search Console から取得したデータが表示されるか確認する。
  - `/gsc-import` でのインポートフローがエラーなく完了し、意図した範囲のデータが取り込まれているか確認する。

- **GA4 連携変更時**
  - GA4 プロパティおよびイベント設定画面で設定内容が正しく反映されるか確認する。
  - 日次同期やイベント集計に関わる変更の場合は、対象期間のメトリクスが期待通りになるかを確認する。

- **スタッフ招待・権限まわり**
  - スタッフ招待ユーザーの参照／削除が期待通り動作するか確認する。
  - オーナーの書き込み不可な領域（ロールや Stripe サブスクリプション ID など）が、閲覧専用オーナー (`role='owner'`) から編集できないことを確認する。

# 代表的なトラブルシューティング

- **LIFF トークンエラー**
  - `authMiddleware` のログを確認し、トークンの有効期限・検証ロジックに問題がないか確認する。
  - `app/api/refresh` エンドポイントの挙動を確認し、トークンリフレッシュが想定通り動作しているかをチェックする。

- **SSE（Server-Sent Events）が途切れる場合**
  - ping 間隔（例: 20 秒）と idle timeout（例: 5 分）の設定が、既存実装と整合しているか確認する。
  - `sendPing` 実装を見直し、一定間隔でクライアントに ping が送られているか、タイムアウト条件が厳しすぎないかを確認する。

- **WordPress 投稿取得に失敗する場合**
  - `WordPressService` の `getRestRequestConfig` 実装を確認し、候補となる REST API URL のリストと認証ヘッダの設定を見直す。
  - 実際にどの URL にリクエストを投げているか、レスポンスステータスやエラーメッセージをログで確認する。

- **Supabase の RLS が原因で操作できない場合**
  - 対象テーブルの RLS ポリシーを、対応するマイグレーションファイル（`supabase/migrations/`）で確認する。
  - 必要に応じて Service Role での実行に切り替えるか、RLS ポリシーを修正する（`supabase-rls` スキルの指針に従う）。

# マイグレーションとロールバック

- 新しいマイグレーションを追加した場合:
  - ローカルで `supabase db push` を実行し、スキーマが期待通り適用されるか確認する。
  - ロールバック方針（元に戻すための SQL や手順）を README や PR 説明にコメントとして必ず残す。
